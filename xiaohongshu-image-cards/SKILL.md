# Xiaohongshu Image Cards

## Overview

Use this skill to turn a long article into a small set of clean, readable Xiaohongshu cards rather than a raw long screenshot. Keep the job split in three parts: decide the editorial cut, generate publish-ready metadata, then render or edit the package locally.

## Workflow

1. Find the source text.
   - Read the note, article, or draft.
   - Gather any local images worth reusing on a card.
   - Prefer the original file over a pasted fragment when both exist.

2. Decide the card stack before rendering.
   - Default to 6-9 cards.
   - Use one cover, 4-7 body cards, and one closing card.
   - Keep one core idea per card.
   - Rewrite for readability; do not preserve paragraph boundaries mechanically.

3. Build a render spec JSON.
   - Create a working folder, usually on the Desktop or next to the source document.
   - Write `cards.json` with the page schema from [references/page-spec.md](./references/page-spec.md).
   - Use short titles, compact bullets, and one strong quote or takeaway per page.
   - Keep copy user-facing. Do not mention interface elements, implementation details, or "this page shows".
   - Do not put production/process notes in any user-facing field. Avoid footer text such as "converted from Obsidian", "generated by AI", "draft version", or "图文排版版" unless the user explicitly asks for attribution.
   - Omit `footerNote` by default. Use it only when the note itself is publishable reader-facing content that strengthens the card.
   - Set `meta.theme` to `auto` by default so the renderer can choose a better visual tone from the content itself.
   - Override `meta.theme` only when the user clearly wants a specific direction such as `product`, `reflection`, `alert`, or `playbook`.
   - Include a top-level `post` object with:
     - final Xiaohongshu title
     - 2-4 alternate titles when helpful
     - Xiaohongshu body copy, normally 200-300 Chinese characters unless the user asks for a shorter caption
     - 3-6 tags

4. Render the cards.
   - Run:
   ```bash
   node /Users/coni/.codex/skills/xiaohongshu-image-cards/scripts/render_cards.js \
     --input /absolute/path/to/cards.json \
     --output /absolute/path/to/output-dir
   ```
   - The script writes PNGs, matching HTML source files, and post copy exports.

5. Edit locally when the user wants manual control.
   - Launch:
   ```bash
   node /Users/coni/.codex/skills/xiaohongshu-image-cards/scripts/edit_cards.js \
     --input /absolute/path/to/cards.json \
     --output /absolute/path/to/output-dir
   ```
   - Add `--open` on macOS or Linux when you want the browser to open automatically.
   - Use the editor to adjust page copy, reorder pages, edit title/body/tags, then click "保存并重新渲染".

6. Visually inspect and tighten.
   - Open at least the cover, a dense middle page, and the closing page.
   - Check whether the chosen theme matches the article tone; when `auto` feels slightly off, first fix copy rhythm before forcing a theme.
   - If a page feels crowded, shorten the lede or reduce bullets before touching font sizes.
   - Remove stray process notes such as "发小红书时..." before final delivery.

7. Deliver paths and brief notes.
   - Return the absolute output folder path and the main PNG files.
   - Return the post copy files too when `post` metadata exists:
     - `xiaohongshu-post.json`
     - `小红书发布文案.md`
   - Mention if the user may need to reopen a file because of Finder preview cache.

## Editorial Rules

- Prefer an image-post rhythm, not an article screenshot rhythm.
- Make the cover hook-led and clean; avoid stuffing it with too many sub-points.
- Keep body cards around:
  - title: 1-2 lines
  - lede: 1-3 sentences
  - bullets: usually 2-4 items
- Use `cover`, `standard`, `insights`, `image`, `workflow`, and `final` page types when possible; they map cleanly to the bundled renderer.
- Use `image` pages only when the visual genuinely helps.
- Keep the voice close to the source, but compress aggressively for mobile reading.
- Keep provenance and workflow notes out of the rendered cards; they belong in the final assistant response, not in the image copy.
- Write the publish caption (`post.body`) as a fuller 200-300 Chinese character body by default: include the hook, the core point, and one soft call-to-action or reflection prompt. Avoid one-sentence captions unless the user explicitly wants a very short version.

## Output Conventions

- Default canvas: `1242x1660` PNG.
- Put rendered PNGs at the output root.
- Put generated HTML under `output-dir/source/`.
- Name pages `01.png`, `02.png`, and so on.
- Put exported post copy at the output root:
  - `xiaohongshu-post.json`
  - `小红书发布文案.md`

## Browser Dependency

The renderer needs a Chromium browser. It checks, in order:

- `CHROME_BIN`
- `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome`
- `/Applications/Chromium.app/Contents/MacOS/Chromium`
- `/Applications/Microsoft Edge.app/Contents/MacOS/Microsoft Edge`
- `/Applications/Brave Browser.app/Contents/MacOS/Brave Browser`

If none exists, stop and tell the user which browser path is missing.
