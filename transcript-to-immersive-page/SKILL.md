# 把长文吸收成网页

这个 skill 的输出不是"会议纪要的 HTML 版"——那是在浪费你的能力。它的输出是 **一份可以钉在墙上、隔三个月再读还能再榨出价值的深度阅读体验**。读者打开它,愿意从头滑到尾,合上之后能复述出三句话、能画出一张图、能在下次开会时引用一段话。

## 为什么要刻意约束自己

LLM 默认行为是向均值收敛——给十次任务,默认会做出十个 Notion 文档、十种"白底+浅蓝按钮+居中卡片"的现代极简风。**这正是要刻意打破的**。这个 skill 存在的意义,在于把每一次输出推向"<u>这一份就是为这一份内容做的</u>",而不是让所有逐字稿都长得像同一个产品的不同章节。

两条核心约束,贯穿全流程:

**1. 反向工程,不是复述。** 你不是在做会议纪要。你在把口语漂浮的方法论、洞察、金句,**反向工程为可复用的工作框架** ——抽出骨架、删掉口水、压成清单、留住关键比喻、加上自检清单 + 金句 Coda。读者读完应该觉得"原文我可以不读了"。

**2. 每次设计根本不同。** 在动手写代码前,**必须**扫一眼 `~/Downloads/` 里已有的同类页(以及任何用户指出的对照样本),记录它们用了什么色板 / 字体 / 形制 / 交互 / 视觉签名。然后**刻意选一组完全不同的**。如果上一份是衬线 + 朱砂红的出版物,这一份就不能再是衬线 + 任何红色;如果上一份是工程蓝图网格,这一份就要换成纯纸张或大色块或全黑。

下面是工作流。每一步都对应一个具体动作,不是清单背诵。

## 工作流 · 8 步

### Step 1. 把整份内容读完

用合适的工具读全文,不要只看前几屏就开干。

| 文件类型 | 读取方式 |
|---|---|
| `.md` / `.txt` | Read tool 直读 |
| `.doc` / `.docx` | `python3 -c "from docx import Document; ..."` 提取纯文本 |
| `.pdf` | Read tool 支持 PDF,大文档加 `pages` 参数分页 |
| `.srt` / `.vtt` | 用 sed/awk 去时间戳,只留正文 |

如果文件超过 30k 字,Read 工具一次读不完——分段读到看清完整结构为止。**读不完就敢开写,基本会跑偏。**

### Step 2. 提炼骨架

读完后,把内容压成 5-8 个章节(超过 9 个就过载,少于 4 个就太薄)。每个章节给三个东西:
- **一个核心论点**(一句话能说清)
- **一个关键证据**(原文里最有力的例子或金句)
- **一个落地动作**(读者读完能做什么)

把这个骨架写在心里(不需要单独存文件),然后**给用户做一行预览** —— 类似:"提炼了 7 个模块:① …② …③ …,主要聚焦 ③。" 等用户确认或纠正方向后再写代码。这一步省下后面 80% 的返工。

### Step 3. 扫已有产物

```bash
ls ~/Downloads/*.html 2>/dev/null | head -20
```

如果有同类页,**快速过一遍**(读 head 部分的 CSS variables 和 fonts link 就够了),记录:
- 用了什么色板(主色 + 强调色)
- 用了什么字体(中英组合)
- 形制是什么(出版物 / 蓝图 / 策略书 / 教科书 / 杂志 / 其他)
- 用了什么独特的视觉签名(噪点 / 网格 / 印章 / 红铅笔批改 / 中英书脊 …)
- 用了什么交互(切换器 / 计算器 / 对照器 / hover / scroll-trigger)

**记住这个清单**——下一步是选**对立面**。

### Step 4. 刻意选定本次设计语言

写代码**前**,在心里(或简短给用户)明确以下五件事:

| 维度 | 必须明确 | 防漂移要点 |
|---|---|---|
| **形制 (form)** | 选定一个有名字的形制 | 见下方形制库;**不要"现代极简"** |
| **色板 (palette)** | 主色 1-2 + 强调色 1-2 + 中性色 | 全部用 OKLCH;主色和强调色饱和度差距要大 |
| **字体 (typography)** | 中文 1-2 + 英文 display 1 + 英文 body 1 + mono 1 | 中英衬线/黑体对仗,避免上一份用过的家族 |
| **视觉签名 (signature)** | 1-3 个独特装饰元素 | 噪点/网格/印章/批改痕迹/滚轮纹/书脊…必须区别于上次 |
| **核心交互 (interaction)** | 至少 1 个真有用的交互 | 切换器/计算器/对照器/滑块,**不是 hover 装饰** |

> 形制是"有名字的"很重要。说"现代极简"不算选择;说"瑞士平面设计的杂志内页"才算。具体名字让你被风格的内在一致性拽住,而不是漂浮。

### Step 5. 写代码

**单 HTML 自包含**,这是硬规则——用户拿到的是一个文件,双击就能看,不需要 build,不需要本地服务器。

技术骨架:
- HTML5 doctype + 中文 lang
- 单 `<style>` 块,所有 CSS 内嵌
- Google Fonts 通过 `<link>` 加载(用户有网即可)
- `<script>` 内嵌(不依赖外部 JS 库;Chart 之类必要时用 CDN)
- CSS 变量管理设计 token(`--paper`, `--ink`, `--accent` 等)
- 适度做响应式(`@media (max-width: 880px)` 至少处理一档)
- 顶部进度条 / 平滑滚动这类基础体验是默认有的
- **目录导航(必备)**:见下方"目录导航 · 硬约束"

#### 目录导航 · 硬约束

每一份产物**必须**有一个常驻的目录导航,不是临时的页内锚点链接,而是**一直在视野里、能告诉读者"我在第几章 / 还剩几章"**的组件。这是长文吸收体验的基础设施,不是可选装饰。

功能要求(任何形态都必须满足):
- **常驻可见**:固定在视口某处(左/右侧栏、顶部吸附、悬浮 dock 都可),不是只在页面顶部出现一次的目录块
- **实时高亮当前章节**:用 `IntersectionObserver` 监听章节进入视口,高亮对应 toc 项(`rootMargin` 调到 `-30% 0px -55% 0px` 一带,只有"主导视口的那个 section"才被高亮)
- **点击平滑滚动**:`scrollIntoView({ behavior: 'smooth', block: 'start' })`
- **覆盖所有顶级节点**:不只是 7 个 Module,还要包括自检清单 / Coda 这种次级落点
- **响应式**:小屏(< 1120px 大致是阈值)折叠成一个可展开的按钮 / 抽屉,不能强占视野
- **样式与形制一致**:出版物形制用衬线 + 边距数字;蓝图形制用 mono + 网格;监控形制用荧光高亮态。**不要硬塞一个通用样式**——目录导航的视觉签名也是形制的一部分

形态可以多样(每次刻意换一种):
- 左侧固定窄栏(本份样本用法,Mission Control 形制)
- 右侧 spine 式数字标记(文学出版物 / 中英书脊形制)
- 顶部吸附 tab bar(教科书 / 蓝图形制)
- 圆形 dock / 角落悬浮章节号(美术馆 catalog 形制)
- 边栏可折叠抽屉(brutalist / 小报形制)

**展开/收起策略也按形制决定,不要每次都做"默认窄态 + hover 展开"**:
- **始终展开**:导航本身是形制的核心视觉(spine 数字 / tab bar / 中文书脊),不应折叠
- **默认收起为章节号,hover/focus 展开**:辅助性窄栏,主体不能被挡(本份样本走的路线 — Mission Control 形制 50px 窄态,hover 后 208px)
- **滚动时浮现,空闲时淡出**:极简形制,最大化阅读沉浸感(美术馆 catalog)
- **永远只显示进度圆点,点击弹出全目录**:高密度极简(brutalist)
- **移动端必折叠**:任何形制,< 1120px 都不能强占主内容视野,改顶部按钮 / 抽屉触发

**反例**:只放一个静态的"目录"段落在页面顶部、点击跳转后没有 active 状态、移动端把窄栏硬塞在主内容左边、与全篇视觉风格脱节的纯灰色列表、**每份产物都用相同的"默认折叠 + hover 展开"动作模式**(那是在重复样本,不是在做形制)。

代码长度参考:**800-1500 行 HTML/CSS/JS**(单文件)。短于 800 行,大概率没把内容嚼透;长于 1500 行,大概率在堆装饰。

### Step 6. 标准结构(可调,但每块通常都该有)

```
[TOC / Index]           常驻目录导航(必备 · 见 Step 5 硬约束)
  ├ 形制一致的视觉签名(不是通用灰色列表)
  ├ 覆盖所有顶级节点(包括 Self-check / Coda)
  ├ IntersectionObserver 实时高亮当前章节
  └ 响应式 · 小屏折叠

[Cover / Masthead]      期刊/封面感,不是导航栏
  ├ 标题(中文为主,英文 display 副标)
  ├ 一段 30-60 字的 deck(把核心冲突压成一段)
  ├ 元数据(来源/时长/章节数/日期)
  └ 形制相关的视觉签名(印章/spine/编号)

[5-8 个 Chapter Modules]
  每个章节:章节号 + meta + 标题 + deck + 主体 + 1-2 个图示/对比/卡片
  ⚠ 每个 Module 都要有可锚定的 id,供 TOC 高亮使用

[1+ Interactive Module]    至少 1 个真有用的交互
  比如:切换器(看同一卖点不同版本)/ 计算器 / 对比器

[Self-check / Takeaways]   带回家的自检清单(5-10 条)
[Coda]                     一句金句 + 出处
[Colophon]                 出处 / 字体 / 设计说明 / 致读者
```

### Step 7. 输出 + 在浏览器打开

写到 `~/Downloads/<descriptive-slug>.html`,文件名描述内容(不是 `output.html`)。用 `open` 在浏览器自动打开:

```bash
open ~/Downloads/<filename>.html
```

> 在 Claude Code 的 launch preview 面板里,文件写完会自动出现在右侧——告诉用户"已可在 preview 面板查看"。

### Step 8. 收尾说明

给用户一段简短的回顾:
- 选了什么形制(为什么)
- 提炼了几个模块,每模块讲什么
- 哪个交互/章节最值得反复用
- 跟之前的产物有什么呼应或差异

不要复述完整内容(那是页面本身的事)——只说"为什么这样设计"。

## 形制库 · 灵感清单

不要把这个清单当菜单,把它当"<u>提醒你形制可以多样</u>"的开口。每选一种就要钻进它的内在一致性。

**编辑/出版物类**
- 文学杂志(The Paris Review / n+1 / The Drift)— 衬线为主,大字号 italic,§ 节标记,边距数字
- 学术期刊— 高密度衬线,脚注感,行号,ABSTRACT/KEYWORDS 框
- 70 年代出版物— OKLCH 暖纸 + 朱砂红,印刷雕版感
- 中文文人出版物— 宋体 + 竖排标题或 spine,§ 配 § 节标记

**工程/技术类**
- 工程蓝图— 暖白底 + 极细网格 + 警示橙 + 等宽编号
- 学术教材— 大字号黑体编号 + 强结构 + 对比卡片
- IDE/代码编辑器— 深色 + monospace + 语法高亮风格的强调
- 数据 dashboard— 但**避开 chart.js 默认**;手画 SVG

**广告/商业类**
- 复古广告策略 deck— 米黄 + 印章红 + 芥末金 + 大字号衬线 BEFORE/AFTER
- Wieden+Kennedy 风— 大字打断 + 反转色块
- 苹果产品页— 大量留白 + 巨幅字体 + 极简

**艺术/文化类**
- 美术馆 catalog— 极简 + 大字标题 + 深 navy
- 日式杂志— 大量留白 + 字体作为主角 + 细线分隔
- 包豪斯几何— 红黄蓝原色 + 几何块
- 中世纪手抄本— 大首字下沉 + 装饰花纹 + 羊皮纸色

**新媒体类**
- VHS / 复古 8-bit— 像素字体 + 扫描线 + 霓虹
- Brutalist Web— 纯文字 + Times 默认 + 黑底白字 + 故意丑
- 杂志小报 tabloid— 红色大字 EXCLUSIVE + 对角排版

**还有更多**——这只是开口。看到内容是哲学思辨,可能选文学出版物;看到是工作方法论,可能选工程蓝图;看到是营销洞察,可能选广告策略书;看到是艺术评论,可能选美术馆 catalog。**让内容选形制,不要让形制找内容**。

## 色板 · 用 OKLCH 不是 HSL

OKLCH (`oklch(L C H)`) 在视觉上更线性,调色板配色更可控。基础写法:

```css
:root {
  --paper:   oklch(0.96 0.02 88);   /* 暖纸 */
  --ink:     oklch(0.20 0.02 30);   /* 深棕黑 */
  --accent:  oklch(0.55 0.18 30);   /* 朱砂红 */
  --gold:    oklch(0.72 0.15 80);   /* 旧金 */
  --mute:    oklch(0.55 0.02 35);   /* 静默灰 */
}
```

**配色原则**:
- 主色 ≤ 4 个(再多就花)
- 主色和背景的 L 差距 ≥ 0.4(否则糊)
- 强调色饱和度(C)要远高于主色(否则没层次)
- 中文衬线在偏暖色系背景上更舒服;黑体在中性 / 冷色系背景上更精神

**对照已用过的色板**(如果同期有兄弟页),换色相区(hue): 已用过 30 度区(红橙)→ 这次试 80 度区(金黄)/ 240 度区(蓝)/ 155 度区(绿)。

## 字体 · 中英对仗,有 mono

全部从 Google Fonts 加载:

```html
<link href="https://fonts.googleapis.com/css2?family=...&display=swap" rel="stylesheet">
```

**几个组合参考(每次至少换一对)**:

| 场景 | 中文 | 英文 display | 英文 body | mono |
|---|---|---|---|---|
| 文学出版物 | Noto Serif SC | Newsreader | Noto Serif | JetBrains Mono |
| 工程蓝图 | Noto Sans SC | Bricolage Grotesque | Inter Tight | JetBrains Mono |
| 广告策略书 | Noto Serif SC | Fraunces | DM Sans | DM Mono |
| 学术教材 | Source Han Serif SC | EB Garamond | Source Sans 3 | IBM Plex Mono |
| 极简日式 | Noto Sans SC | Cormorant | DM Sans | Space Mono |
| 美术馆 catalog | Noto Serif SC | Playfair Display | Lora | Space Mono |

**避免用过的就换**——每次至少换 display 字体,中文也尽量换(serif/sans 翻转一次)。

## 必须避开的 AI slop

每次写完,自己回头检查这张反例清单。**任何一条命中,改稿。**

- ❌ 紫色渐变在白底上(最常见 AI 风)
- ❌ "Built with ❤️" / "Get Started" / "Learn More" 这类英文 SaaS 套话
- ❌ Inter / Roboto / Arial 等"安全选择"字体
- ❌ Material Design / shadcn 默认的圆角卡片 + 阴影
- ❌ Hero + 三栏 features + CTA + footer 的"落地页"结构(除非内容真是落地页)
- ❌ 无意义的 emoji 装饰(✨🚀💡)
- ❌ 没有任何视觉签名 / 噪点 / 装饰 —— 干净到没特征
- ❌ 把章节标题写成 "01. The First Chapter" 这种 AI 味命名,而不是从内容里长出来的标题
- ❌ 配色都用 60% 灰度 + 一个蓝色 accent
- ❌ 文字密度均匀分布,没有大字号视觉锚点
- ❌ 交互是装饰性 hover,不是真有信息密度的对照器/计算器/切换器
- ❌ 把口语会议直接搬上去(违反"反向工程"原则)

## 内容反向工程的具体动作

把口语逐字稿压成"可读"的几条具体动作:

| 动作 | 解释 |
|---|---|
| **删口水** | "嗯/啊/那个/我说/这个事吧"——全删。"扣个一""听懂了吗"这种互动也删。 |
| **保金句** | 原文里说得最狠最准的那一句话,**原话保留**(用 pull-quote 放大)。 |
| **拆颗粒** | 一段 800 字的口语,通常藏着 3-5 个独立小论点——拆开,每个变成一个小标题或卡片。 |
| **造对照** | 培训类内容必有 before/after 或 反例/正例——把它们对开做。 |
| **抽公式** | 讲师反复用的判断标准,通常能压成一行公式(如"利益点 + 冲突点 = 主标题")。 |
| **加自检** | 全篇看完后,把"读者下次该怎么用"压成 5-8 条 checklist 放结尾。 |
| **找比喻** | 全篇最有力的一个比喻(如"父母把孩子当分红股"),围绕它做一个交互或图示。 |

---

## 交互的判断标准

页面里**至少有一个真有用的交互**——不是 hover 变色,是按动后改变信息的交互。判断"真有用"的标准:

- 它让"反常识"变得**可触摸**(分红计算器把"不用一笔还清"变成可拖动)
- 它让"对比"变得**可切换**(颗粒度切换器一键看三档差异)
- 它让"抽象概念"变得**可演示**(MBTI 介绍 vs 包装的对照器)

如果想不到,就去找原文里的核心比喻或核心反差,把它做成可点击的对照。
